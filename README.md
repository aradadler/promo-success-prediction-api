# Promotion Success Prediction API

## Overview

A production-style ML API that predicts promotion success and translates model output into actionable business recommendations.

This project is a production-minded machine learning system for a common retail planning question: **is a promotion likely to succeed before it goes live?**

In this project, a promotion is considered successful when **ROI is at least 20%**. That business definition is converted into a supervised classification problem, allowing a model to estimate promotion success from inputs that are known before launch.

The result is a portfolio project that connects business logic, synthetic data design, leakage-safe feature engineering, baseline model training, artifact persistence, and a live FastAPI inference layer for real-time scoring. Inference uses the persisted model artifact to keep training and serving aligned.

## Key Highlights

- End-to-end ML system: data -> training -> persisted model -> live API inference
- Decision-support API (not just prediction): probability, label, confidence, interpretation, recommendation
- Leakage-safe feature design using only pre-promotion inputs
- Business-aligned target: ROI >= 20%
- FastAPI-based real-time scoring with validated request/response schemas

## Business Problem / Objective

Retail teams make promotion decisions before they know the outcome. They have to choose discount levels, duration, pricing strategy, and expected volume with limited visibility into whether the campaign will actually create incremental value.

This project addresses that decision point directly. Instead of analyzing promotions only after the fact, it asks whether a promotion is likely to meet a business threshold **before launch**.

The business rule is straightforward:

```text
success = 1 if ROI >= 20%
success = 0 otherwise
```

That turns a business KPI into a machine learning target the system can train on, evaluate, and serve through an API.

In this project, ROI is the return generated by a promotion relative to the profit impact of running it. In plain terms, it asks: did the promotion create enough incremental value to justify the discount and margin tradeoff?

```text
ROI = incremental profit / promotional investment
```

For a business reader, the practical takeaway is simple: a higher ROI means the promotion generated stronger returns relative to what the business gave up to run it.

## Core ML Framing

The modeling task is:

```text
P(success | pre-promotion inputs)
```

Where:

- `success = 1` when ROI is at least 20%
- `success = 0` otherwise

This is a binary classification problem built around a business outcome. The goal is not to predict every simulated intermediate value, but to estimate whether a planned promotion is likely to clear a meaningful profitability threshold.

## System Workflow

The current workflow is structured as a simple, production-style ML pipeline:

1. Generate a synthetic promotion dataset with realistic retail planning inputs.
2. Simulate promotion outcomes such as uplift, expected units, profit, and ROI.
3. Create a binary success label from the ROI threshold.
4. Build leakage-safe model features from only the inputs available before launch.
5. Train and evaluate a baseline classification model.
6. Persist the trained artifact to `models/model.pkl`.
7. Load the saved model artifact and serve live predictions through FastAPI.

In practical terms, the workflow is: data -> feature engineering -> model training -> saved artifact -> inference API. That keeps the system consistent from offline training to live decision support.

## Data Design

The dataset is intentionally designed around what a retailer could know at planning time versus what only becomes visible after a campaign is executed.

### Approved Raw Inputs

These are the inputs the model is allowed to see:

- `price`
- `discount_pct`
- `baseline_units`
- `cogs`
- `cannibalization_pct`
- `duration_weeks`

### Synthetic Data Generation

This project does not use real retailer data. Instead, the dataset was intentionally simulated to reflect realistic pre-promotion planning variables such as price, discount, baseline demand, cost of goods, cannibalization, and promotion duration.

From those simulated inputs, the project then generated outcome variables such as uplift, expected units, profit, and ROI using business logic. The goal was to create a realistic sandbox for modeling, feature design, and API development, not to claim production-ready accuracy on real commercial data.

As one simple example, baseline demand was approximated through simulated demand points rather than actual retailer transaction history.

### Engineered Model Features

These features are derived from approved pre-promotion inputs and are used by the model:

- `promo_margin_per_unit`
- `discount_to_margin_ratio`
- `baseline_profit`

### Label Creation

Outcome simulation produces ROI, and the label is then created as:

```text
success_label = 1 if roi >= 0.20 else 0
```

This preserves a clean separation between business outcome generation and model training.

## Leakage Prevention

Feature leakage is one of the most important modeling risks in this project. If the model were allowed to train on values that are only known after a promotion runs, it would appear strong in training but fail in a real planning workflow.

This project avoids that issue by restricting training inputs to pre-launch information only.

### Included In The Model

Approved raw inputs:

- `price`
- `discount_pct`
- `baseline_units`
- `cogs`
- `cannibalization_pct`
- `duration_weeks`

Engineered model features:

- `promo_margin_per_unit`
- `discount_to_margin_ratio`
- `baseline_profit`

### Explicitly Excluded From The Model

These fields exist for simulation and label generation, but are not allowed into training features:

- `uplift_pct`
- `expected_units`
- `incremental_units`
- `promo_profit`
- `incremental_profit`
- `roi`
- `baseline_profit_reference`

This design makes the model more credible as a real pre-launch decision-support system.

## Model Training

The current model is a Logistic Regression baseline built with a scikit-learn Pipeline using `StandardScaler` and `LogisticRegression`. This is an intentional choice: it is fast to train, easy to interpret, and a strong baseline before moving to more complex models.

The model predicts the probability that a planned promotion will be successful, where success means clearing the business rule of ROI >= 20%.

The training dataset contains about `10,000` simulated promotions. Class balance is roughly `79%` non-success and `21%` success, which provides a more realistic evaluation context than a perfectly balanced synthetic dataset and helps frame how to interpret model performance.

## API Inference Output

The API does more than return a raw model score. It translates the prediction into business-facing decision support so a planner, PM, or analyst can quickly understand whether a proposed promotion looks viable.
It goes beyond raw probabilities by layering decision logic, confidence, and recommendations around the model output for business users.

- `success_probability`: the model's estimated probability that the promotion will meet the ROI success threshold.
- `predicted_label`: a simple `SUCCESS` or `FAILURE` classification derived from the probability.
- `confidence`: a qualitative confidence band (`HIGH`, `MEDIUM`, or `LOW`) that indicates how decisive the score is.
- `interpretation`: a plain-English summary of what the probability means.
- `recommendation`: an action-oriented suggestion that helps turn the model output into a practical next step.

### Sample Request

```json
{
  "price": 10.0,
  "discount_pct": 0.2,
  "baseline_units": 100,
  "cogs": 5.0,
  "cannibalization_pct": 0.1,
  "duration_weeks": 2
}
```

### Sample Response

```json
{
  "success_probability": 0.92,
  "predicted_label": "SUCCESS",
  "confidence": "HIGH",
  "interpretation": "High likelihood of success",
  "recommendation": "Promotion is likely viable - consider proceeding"
}
```

For example, a pricing or promotion manager could test different discount and duration scenarios before launch to compare which plan looks most likely to meet the ROI threshold.

## Model Results

Baseline model results from the current implementation:

- Dataset size: `10,000` rows — number of simulated promotions used for training
- Class balance: `{0: 0.7857, 1: 0.2143}` — proportion of unsuccessful vs successful promotions
- Accuracy: `0.9685` — overall correctness of predictions
- Precision: `0.9507` — how often predicted “successful” promotions are actually successful
- Recall: `0.8998` — how many truly successful promotions the model correctly identifies
- F1: `0.9246` — balance between precision and recall
- ROC-AUC: `0.9938` — overall ability to distinguish successful vs unsuccessful promotions

From a business perspective, precision helps avoid recommending poor promotions, while recall ensures strong opportunities are not missed. Together, these metrics indicate that the model can support promotion planning decisions with high confidence.

These results are strong for a baseline model on synthetic data and show that the workflow is behaving coherently. Some of the high performance is expected because the data was generated from structured business rules rather than messy real-world retail behavior. Validating the approach on real-world retail data would be a critical next step. The next stage is to harden the training and inference path rather than treat the current score as the final goal.

## How to Run

### Generate The Dataset

If you need to regenerate the synthetic training data:

```bash
python data/generate_data.py
```

This creates `data/promotions.csv` by generating base inputs, simulating promotion outcomes, and assigning `success_label` from ROI.

### Train The Model

Run the baseline training pipeline with:

```bash
python -m src.app.model
```

That command:

- loads `data/promotions.csv`
- builds the approved leakage-safe feature set
- splits the data into train and test sets
- trains a Logistic Regression baseline
- evaluates performance using standard classification metrics
- saves the trained artifact to `models/model.pkl`

If your local environment uses `python3` instead of `python`, use the equivalent command with `python3`.

### Running The API Locally

1. Activate your conda environment.
2. Install dependencies if needed.
3. Start the API:

```bash
uvicorn app.main:app --reload
```

If you are running from the repository root, use `--app-dir src` or set `PYTHONPATH=src` so the `app` package resolves correctly.

- Swagger UI: `http://127.0.0.1:8000/docs`
- Health check: `http://127.0.0.1:8000/health`

Swagger UI lets you test the `/predict` endpoint directly from the browser.
